# LinguaFlow - Full Application Improvement Plan

## Executive Summary

This plan outlines a 6-phase roadmap to transform LinguaFlow from an MVP to a production-grade Japanese learning app. The app currently has basic functionality but suffers from architectural issues, minimal testing, and missing features.

---

## Phase 1: Foundation & Architecture Refactoring

### Goals
- Establish proper Clean Architecture
- Implement Dependency Injection (Hilt)
- Split monolithic ViewModel into feature-specific ViewModels
- Set up Navigation Component

### Technical Approach
1. Add Hilt dependency and setup
2. Create feature-specific ViewModels:
   - `LearnViewModel` - Learn tab logic
   - `PracticeViewModel` - Practice tab logic
   - `DictionaryViewModel` - Dictionary/search logic
   - `SpeechViewModel` - Speech practice logic
   - `ProfileViewModel` - Profile/stats logic
3. Create Use Cases for business logic
4. Implement Navigation Component with proper graph
5. Create navigation-safe arguments

### Files to Modify/Create
- `build.gradle.kts` - Add Hilt dependencies
- `app/src/main/java/com/example/LinguaFlowApplication.kt` - Hilt application class
- `app/src/main/java/com/example/ui/navigation/NavGraph.kt` - Navigation graph
- `app/src/main/java/com/example/ui/navigation/NavRoutes.kt` - Route constants
- `app/src/main/java/com/example/domain/usecase/` - Directory for use cases
- `app/src/main/java/com/example/ui/viewmodel/*` - Split ViewModels
- `MainActivity.kt` - Replace custom navigation with NavHost

### Dependencies to Add
```kotlin
implementation(libs.hilt.android)
implementation(libs.hilt.navigation.compose)
ksp(libs.hilt.compiler)
```

### Testing Strategy
- Unit tests for each ViewModel
- Navigation tests
- Hilt integration tests

### Estimated Effort
- 2-3 days

## Phase 2: Data Layer & Storage Improvements

### Goals
- Fix architectural violations
- Implement proper database migrations
- Add offline-first strategy
- Improve repository patterns

### Technical Approach
1. Separate `SpeechHistory.kt` entity from DAO
2. Create `SpeechHistoryDao.kt`
3. Add missing migrations:
   - `MIGRATION_1_2` - Create practice_history and speech_history tables
   - `MIGRATION_2_3` - Already exists, verify correctness
4. Implement Repository pattern with DataSource abstraction
5. Add network mapper for API responses

### Files to Modify/Create

- `app/src/main/java/com/example/data/Migration1To2.kt` - New migration

### Testing Strategy
- Database migration tests

- Repository tests

- 2-3 days

## Critical Path Dependencies

1. **Phase 1** must complete before other phases (architecture foundation)
2. **Phase 2** depends on Phase 1 (Hilt DI needed for repositories)
3. **Phase 3** depends on Phase 1 (Navigation Component)
4. **Phase 4** can partially run in parallel with Phase 3
5. **Phase 5** should start after Phase 1-4 for proper testing setup
6. **Phase 6** requires all previous phases

---

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Breaking changes during refactoring | Use feature branches, incremental commits |
| Firebase integration issues | Use Firebase v33+ with proper setup |
| Performance degradation | Profile early, use benchmarking |
| Test flakiness | Use deterministic tests, avoid timing issues |
| API key exposure | Use local properties, never commit keys |
